### 前情学习

#### 类的关系

#### 1. **继承（Inheritance）**

- **关系语义**：is-a关系（子类继承父类属性和方法）

- **代码特征**：强耦合，父类变化影响子类（如`Dog extends Animal`）。

- **Mermaid语法**

  ```mermaid
  classDiagram
      Animal <|-- Dog : 继承
  ```

  

  **箭头方向**：子类指向父类（

  ```
  <|--
  ```

------

#### 2. **实现（Realization）**

- **关系语义**：can-do关系（类实现接口方法）

- **代码特征**：松耦合，接口定义方法签名（如`Bird implements Flyable`）。

- **Mermaid语法**

  ```mermaid
  classDiagram
      Flyable <|.. Bird : 实现
      class Flyable {
          <<interface>>
          fly()
      }
  ```

  

  **箭头方向**：实现类指向接口（

  ```
  <|..
  ```

  或

  ```
  ..|>
  ```

------

#### 3. **关联（Association）**

- **关系语义**：has-a关系（类间长期引用，生命周期独立）

- **代码特征**：成员变量持有对象（如`Teacher`持有`Student`列表）。

- **Mermaid语法**：

> [!IMPORTANT]
>
> 关联-->聚合-->组合的拥有关系是递进的
>
> 其表现形式**都是成员变量**
>
> 例子如下：
>
> - 关联：教师--学生
> - 聚合：车--轮子
> - 组合：人--大脑
>
> 教师拥有一群学生，学生也拥有共同的老师，但他并不是“所有”的感觉。去除这层关系，他们是独立的个体。
>
> 车子拥有四个轮子，但是轮子是可以替换的，轮子坏了，换了轮子，车依旧是好的。也就是轮子拥有独立的生命周期。
>
> 人拥有大脑，大脑死了，人就死了，

### 访问者模式

> 根据身份的不同，每个人员的访问内容的进行的操作也不同
>
> 面对一个**由许多不同类型的对象组成的复杂对象结构**（例如，树形结构、组合结构），并且需要**对该结构中的所有元素执行多种不同操作**时 使用访问者模式

1. 如果一个数据类要满足所有访问者的需求，就需要**新增很多”访问“方法**。会让数据类变得十分冗余。
2. 并且每个数据类都有不同的“访问”方法，每次调用都需要找到相应的数据类的访问方法，操作逻辑分散



访问者模式就是解决这些问题：

1. **新增访问者角色**：每个角色拥有独立的操作数据的方法，将数据类的访问方法解偶出来
2. **新增对象结构**：管理各个具体数据类，通过list（抽象元素类），“接待”访问者完成**所有**元素的访问

```mermaid
sequenceDiagram
    participant Client
    participant ObjectStructure
    participant ConcreteElementA
    participant ConcreteVisitor
    Client->>ObjectStructure: accept(&visitor)
    ObjectStructure->>ConcreteElementA: accept(visitor)
    ConcreteElementA->>ConcreteVisitor: visit(this)
    ConcreteVisitor->>ConcreteElementA: operationA()
```

- ✅ 操作扩展性：添加新操作只需新增访问者类（无需修改元素类）
- ✅ 数据结构稳定：元素类结构变化频率较低时最有效
- ⚠️ 元素扩展成本：新增元素类型需修改所有访问者类

### 观察者模式

> 观察者模式（Observer Pattern）是一种**行为型设计模式**，用于建立对象间一对多的依赖关系，使得当一个对象（被观察者/主题）状态改变时，其所有依赖对象（观察者）会自动收到通知并更新

```mermaid
sequenceDiagram
    participant Subject
    participant ObserverA
    participant ObserverB
    
    Note over ObserverA,ObserverB: 注册阶段
    ObserverA->>Subject: attach()
    ObserverB->>Subject: attach()
    
    Note over Subject: 状态变更阶段
    Subject->>Subject: setState(newValue)
    Subject->>Subject: notify()
    
    Note over ObserverA,ObserverB: 通知阶段
    Subject->>ObserverA: update()
    Subject->>ObserverB: update()
    
    Note over ObserverA: 执行响应逻辑
    Note over ObserverB: 执行响应逻辑
```

**模式变体与对比**

1. **推模型 vs 拉模型**
   - 推模型：主题主动将数据传给观察者（如update(temperature)）
   - 拉模型：观察者从主题中主动拉取数据（需主题暴露获取方法getState）
2. **与发布-订阅模式区别**
   - 观察者模式：**直接通信**，主题**持有观察者引用**（因为需要观察者的update方法）。
   - 发布-订阅模式：**通过消息代理间接通信**，彻底解耦（如Redis Pub/Sub）



**优点**

- 松耦合架构：主题不关心观察者具体实现
- 动态订阅：运行时增减观察者
- **通信灵活**：推/拉模型适应不同数据需求

**缺点**

- 通知notify其实是循环执行各个观察者的update，存在性能浪费。

### 模版方法模式

> **模板方法模式**是一种**行为型设计模式**，通过定义算法的骨架（固定步骤），将某些步骤的具体实现延迟到子类中。

```mermaid
sequenceDiagram
    participant Client
    participant AbstractClass
    participant ConcreteClass
    Client->>AbstractClass: 调用 templateMethod()
    AbstractClass->>AbstractClass: 执行 step1()（固定步骤）
    AbstractClass->>ConcreteClass: 调用 step2()（子类实现）
    ConcreteClass-->>AbstractClass: 返回结果
    AbstractClass->>AbstractClass: 执行 step3()（固定步骤）
    AbstractClass-->>Client: 返回最终结果
```

| **优点**                                   | **缺点**                                                 |
| :----------------------------------------- | :------------------------------------------------------- |
| ✅ **代码复用**：公共逻辑集中父类，避免重复 | ❌ **继承强耦合**：子类依赖父类，修改父类可能影响所有子类 |
| ✅ **扩展灵活**：新增子类即可支持新行为     | ❌ **类膨胀**：每个变体需单独子类，导致类数量过多         |
| ✅ **流程标准化**：确保核心逻辑不可变       | ❌ **灵活性受限**：子类无法修改算法结构（如步骤顺序）     |

### 适配器模式

> 适配器模式（Adapter Pattern）是一种**结构型设计模式**，用于解决接口不兼容问题，使原本无法协同工作的类能够协同工作。

1. **目标接口（Target）**
   - 客户端期望的接口规范，定义需实现的方法
2. **被适配者（Adaptee）**
   - 已存在的类或接口，但其接口与目标接口不兼容
3. **适配器（Adapter）**
   - 实现目标接口，并封装被适配者的实例，在目标方法中调用被适配者的方法

### 外观模式

> 外观模式是一种**结构型设计模式**，旨在为复杂的子系统提供一个统一的接口，隐藏内部细节，降低客户端与子系统的耦合度。其核心思想是：
>
> - 封装复杂性：通过一个高层接口（外观类）封装多个子系统的交互逻辑
> - **简化调用**：客户端仅需与外观类交互，无需直接调用多个子系统的接口

```mermaid
classDiagram
    class Facade {
        +operation()
    }
    class SubsystemA {
        +operationA()
    }
    class SubsystemB {
        +operationB()
    }
    class SubsystemC {
        +operationC()
    }
    
    Facade --> SubsystemA : 调用
    Facade --> SubsystemB : 调用
    Facade --> SubsystemC : 调用
    Client --> Facade : 依赖
```

1. Facade`（外观角色）
   - 提供统一入口，封装子系统的调用逻辑
2. `Subsystem`（子系统角色）
   - 实现具体功能（如CPU;Memory;Light等），对外界透明
3. `Client`（客户端）
   - 通过外观类调用功能，不直接接触子系统

### 享元模式

> 享元模式（Flyweight Pattern）是一种**结构型设计模式**，通过共享对象来减少内存占用、优化性能，适用于系统中存在大量相似对象的场景。
>
> 其核心思想是将对象状态分为**内部状态**（可共享、不变）和**外部状态**（不可共享、可变），通过复用内部状态相同的对象来降低资源消耗

```mermaid
classDiagram
    class FlyweightFactory {
        -pool: Map<String, Flyweight>
        +getFlyweight(key: String): Flyweight
    }
    class Flyweight {
        +operation(extrinsicState: String)
    }
    class ConcreteFlyweight {
        -intrinsicState: String
        +operation(extrinsicState: String)
    }
    class Client {
        -extrinsicState: String
    }
    
    FlyweightFactory o--> Flyweight : 创建/管理
    Flyweight <|.. ConcreteFlyweight
    Client --> FlyweightFactory : 请求对象
    Client --> Flyweight : 传入外部状态
```

1. 抽象享元（Flyweight）
   - 定义操作接口，声明方法参数接收外部状态
2. 具体享元（ConcreteFlyweight）
   - 实现接口，存储内部状态（如 String fontName），处理外部状态
3. 享元工厂（FlyweightFactory）
   - 维护享元池（如HashMap），按需创建或返回现有对象（核心复用逻辑）。
4. 客户端（Client）
   - 存储外部状态，调用工厂获取享元对象并**传入外部状态**

### 原型模式

> **原型模式**是一种**创建型设计模式**，通过复制现有对象（原型）创建新对象，避免重复初始化过程。其核心思想是：
>
> - 对象复用：通过克隆（clone()）替代构造函数调用，提升性能并简化对象创建。
> - 动态扩展：支持运行时动态添加或修改对象类型，无需依赖子类继承。
> - 分离创建逻辑：客户端无需了解对象构造细节，仅通过原型接口获取新实例
>
> **关键术语**：
>
> - 浅拷贝：复制对象本身及基本类型字段，但引用类型字段共享（仅复制引用地址）
>
> - 深拷贝：递归复制对象及其所有引用对象，实现完全独立的新对象

```mermaid
classDiagram
    class Prototype {
        <<interface>>
        +clone(): Prototype
    }
    class ConcretePrototype {
        -field: String
        +clone(): Prototype
    }
    class Client {
        +operation()
    }
    Prototype <|.. ConcretePrototype
    Client --> Prototype
```

**抽象原型（Prototype）**

- 定义克隆接口（如clone()方法），通常为抽象类或接口

### 责任链模式

> **责任链模式**是一种**行为型设计模式**，通过将多个处理对象连成一条链，使请求沿链传递直至被处理。其核心思想是：
>
> - 解耦请求与处理：请求发送者无需知道具体处理者，只需将请求提交到**链首**
> - **动态组合处理逻辑**：可灵活调整链中处理者的顺序或增减节点，符合开闭原则
>
> **类比**：公司请假流程（员工→主管→经理→HR），每个审批者根据权限决定处理或传递请求

```mermaid
classDiagram
    class Handler {
        <<abstract>>
        -nextHandler: Handler
        +setNext(handler: Handler)
        +handleRequest(request: Request)
    }
    class ConcreteHandlerA {
        +handleRequest(request: Request)
    }
    class ConcreteHandlerB {
        +handleRequest(request: Request)
    }
    class Client {
        +submitRequest()
    }
    Handler <|-- ConcreteHandlerA
    Handler <|-- ConcreteHandlerB
    Client --> Handler
```

| **优点**                                                   | **缺点**                                                   |
| :--------------------------------------------------------- | :--------------------------------------------------------- |
| ✅ **解耦请求与处理**：发送者无需知悉处理细节。             | ❌ **性能开销**：长链传递增加响应时间。                     |
| ✅ **灵活扩展**：动态增删处理节点，符合开闭原则。           | ❌ **调试困难**：请求路径复杂，问题定位耗时。               |
| ✅ **责任单一化**：每个处理者专注自身逻辑（如校验、审批）。 | ❌ **请求丢失风险**：若**无兜底处理者**，请求可能未被处理。 |

### 中介者模式

> 中介者模式（Mediator Pattern）是一种**行为型设计模式**，通过引入一个中介对象来封装多个对象间的复杂交互，将网状耦合结构转化为**星型结构**，从而降低系统复杂度。
>
> - 解耦对象交互：将多对多的对象通信转化为一对多关系（中介者与各对象交互），避免对象间直接引用。
> - **集中控制逻辑**：中介者封装协调逻辑，对象只需与中介者通信，无需知晓其他对象细节，符合**迪米特法则**
>
> 智能家居

```mermaid
sequenceDiagram
    participant A as ConcreteColleagueA
    participant M as ConcreteMediator
    participant B as ConcreteColleagueB
    A->>M: 发送消息
    M->>B: 转发消息
    B-->>M: 处理结果
    M-->>A: 返回响应
```

|      角色      | 职责                                                         |
| :------------: | :----------------------------------------------------------- |
| **抽象中介者** | 定义通信接口，管理同事对象的注册与消息转发（如 `Mediator` 接口） |
| **具体中介者** | 实现协调逻辑，维护**同事对象集合**（如 `ConcreteMediator`）  |
| **抽象同事类** | 定义与中介者的通信方法（如 `Colleague` 类）                  |
| **具体同事类** | 实现业务逻辑，通过中介者与其他同事通信（如 `Alarm`、`TV` 等） |

### 装饰模式

> 装饰模式（Decorator Pattern）是一种**结构型设计模式**，通过动态组合而非继承的方式扩展对象功能，保持代码的灵活性和可扩展性。
>
> **场景：咖啡馆点单系统**
>
> - 基础构件：Coffee（基础咖啡，价格10元）。
> - **装饰器**：`MilkDecorator`（加牛奶，+2元）、`ChocolateDecorator`（加巧克力，+3元）

```mermaid
classDiagram
    class Component {
        +operation()
    }
    class ConcreteComponent {
        +operation()
    }
    class Decorator {
        -component: Component
        +operation()
    }
    class ConcreteDecoratorA {
        +operation()
        +addedBehavior()
    }
    class ConcreteDecoratorB {
        +operation()
        +addedBehavior()
    }
    Component <|-- ConcreteComponent
    Component <|-- Decorator
    Decorator <|-- ConcreteDecoratorA
    Decorator <|-- ConcreteDecoratorB
    Decorator o-- Component
```

### 状态模式

> 状态模式（State Pattern）是一种**行为型设计模式**，允许对象在其内部状态改变时改变其行为，使对象的行为看起来像是修改了它的类

```mermaid
classDiagram
    class Context {
        -state: State
        +setState(state: State)
        +request()
    }
    class State {
        <<interface>>
        +handle(context: Context)
    }
    class ConcreteStateA {
        +handle(context: Context)
    }
    class ConcreteStateB {
        +handle(context: Context)
    }
    Context o--> State
    State <|.. ConcreteStateA
    State <|.. ConcreteStateB
```

| **优点**                                                     | **缺点**                                                     |
| :----------------------------------------------------------- | :----------------------------------------------------------- |
| ✅ **消除条件分支**：避免复杂的`if-else`逻辑，代码更清晰。    | ❌ **类数量增加**：每个状态需一个类，可能过度设计。           |
| ✅ **易于扩展**：新增状态只需添加类，无需修改上下文（符合开闭原则）。 | ❌ **状态转换耦合**：状态类需持有上下文引用，可能引入循环依赖。 |
| ✅ **集中状态逻辑**：每个状态的行为局部化，便于维护。         | ❌ **性能开销**：频繁状态切换可能增加对象创建成本（可结合享元模式优化）。 |

### 组合模式

>  组合模式（Composite Pattern）是一种**结构型设计模式**，用于将对象组合成树形结构以表示“部分-整体”的层次关系，使客户端能够以统一的方式处理单个对象和组合对象

```mermaid
classDiagram
    class Component {
        +operation()
        +add(Component)
        +remove(Component)
    }
    class Leaf {
        +operation()
    }
    class Composite {
        -children: List<Component>
        +operation()
        +add(Component)
        +remove(Component)
    }
    Component <|-- Leaf
    Component <|-- Composite
    Composite o-- Component
```

| **角色**                  | **职责**                                                     |
| :------------------------ | :----------------------------------------------------------- |
| **抽象组件（Component）** | 声明所有对象的通用接口（如`operation()`），可定义默认行为（如管理子组件的方法）。 |
| **叶子节点（Leaf）**      | 树形结构的末端节点，无子节点，实现具体业务逻辑（如文件显示、员工信息）。 |
| **组合节点（Composite）** | 存储子组件集合，实现管理子组件的方法（如`add()`、`remove()`），递归调用子组件操作。 |
